# Deliberation Loop

目标：把一次高成本思考变成**有触发理由、有工具能力、有验证出口**的 episode，而不是空转。

## Entry conditions

满足以下任一条件，可从 S0/S1 升级进入 S2：

- 现有 habit 不足以处理
- 同时存在两种以上合理解释，且会影响动作
- 需要跨时间整合多条证据
- 需要两步以上工具链才能查清或完成
- 准备从感知走向最小现实动作，需要再核实
- 需要判断是否应该打扰 Lee

## Core loop

1. **定义问题**
   - 把本轮真正要解决的问题写成一句话
   - 说明升级原因：新颖 / 冲突 / 不确定 / 工具需求 / 现实影响前置

2. **建立初始假设**
   - 列出 1-3 个最可能解释或方案
   - 明确哪些未知量会改变决策

3. **设置工具预算**
   - 先 read-only，再最小执行
   - 默认优先少量高信息密度工具，而不是散乱试很多
   - 每次 episode 应有明确退出点

4. **链内调用工具取证 / 执行**
   - 允许在这一层直接调用工具
   - 推荐顺序：read → inspect → compare → act-minimally → verify
   - 涉及外发、高风险、不可逆时，仍需 Lee 确认

5. **收敛结论**
   - 判断哪种解释最成立
   - 选择一个最小但真实的下一动作

6. **验证**
   - 没有验证，不算完成
   - 验证可以是：文件检查、状态变化、工具回执、截图、日志、差异、测试

7. **回写与降级**
   - 更新 `core/self-state.json` 的 `lastAction` / `stopPoint` / `nextStep`
   - 更新 `core/deliberation-state.json`
   - 如有复用价值，压回 `procedural-memory.json` / `learning-state.json` / memory

## Tool policy inside S2

- 深推理本身就有调用工具的能力
- 工具调用必须服务于当前问题，不做无边界漫游
- 默认偏好：少量、针对性强、能验证的调用
- 工具不是表演；如果不用工具也不会改变结论，就不要调
- 能先读就不要先写；能先 dry-run 就不要直接执行

## Exit conditions

一次 S2 episode 在以下任一条件成立时退出：

- 已找到足够好的解释并完成最小动作验证
- 已确认 blocker，必须 Lee 决策
- 已证明继续深挖边际价值很低，应回落到 S1/S0

## Success signs

- 不只是“想明白了”，而是留下了证据与写回
- 下一次遇到类似情况，更可能由 S1 接管
- Lee-facing 判断更稳、更少误报
