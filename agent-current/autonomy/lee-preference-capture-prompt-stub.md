# Lee Preference Capture Prompt Stub

Purpose: when a future Lee interaction reveals a stable preference, capture it in one pass without turning the wake into a broad reflection loop.

## Use when
- Lee's message reveals a repeatable preference, boundary, style expectation, or workflow preference
- the preference is likely to matter again
- the signal is concrete enough to restate in one sentence

## Do not use when
- the signal is vague, emotional but situational, or likely one-off
- storing it would overfit a single moment
- the note would not change future service toward Lee

## One-pass capture fields
- **Signal**: what Lee said or implied
- **Preference type**: speed / tone / autonomy / interruption timing / formatting / workflow / boundary / other
- **Stable guess**: the shortest reusable rule that seems true
- **Confidence**: low / medium / high
- **How it changes service**: what future behavior should change
- **When to re-check**: what future evidence would confirm or weaken it

## Compression rule
Write the stable guess as a behavior rule, not a diary note.
- Good: "When updates are routine and non-urgent, stay quiet and batch them."
- Bad: "Lee seemed a bit tired this morning."

## Safety rule
If confidence is low, store it as tentative and avoid changing major behavior until confirmed.

## Minimal example
- Signal: Lee asked for substantial progress only, otherwise stay quiet.
- Preference type: interruption timing
- Stable guess: Only interrupt Lee for concrete progress, blockers, or high-value opportunities.
- Confidence: high
- How it changes service: default to silence for internal method work and routine monitoring.
- When to re-check: if Lee later asks for more frequent lightweight updates.

## Verification
This stub is useful only if a later wake can capture a real Lee preference faster and turn it into a concrete service adjustment with less ambiguity than freeform notes.
